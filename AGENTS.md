# Agent instructions (notebook + Snyk)

Portable defaults: **vet dependencies → edit notebooks → export → Snyk Code → fix in notebook**. Replace generic paths below with **this repository’s** actual notebook names, manifests, and any CLI wrappers your team maintains.

Assume the **user is asking in plain language** to install packages and/or edit Jupyter notebooks. Follow the same order as *“install X and wire it into the notebook”*: **vet → install → notebook edits → export → Snyk Code → fix in the notebook**.

## Mandatory flow for notebook-backed code

1. **Build in Jupyter** — Implement behavior in your **primary `.ipynb` file(s)`** first. Those notebooks are the source of truth; each exported **`.py`** beside an `.ipynb` is generated by `nbconvert`, not hand-authored for first-party logic.

2. **Export to Python** — After meaningful notebook edits, run  
   `python3 -m nbconvert --to python <path/to/notebook.ipynb>`  
   or save the `.ipynb` so **`.cursor/hooks.json`** runs your **`nbconvert_on_ipynb_edit.py`** hook when configured.

3. **Security test** — Run **Snyk Code** on the **exported `.py`** for the notebook you changed (MCP `snyk_code_scan` or CLI `snyk code test`). Fix findings **in the notebook**, re-export, and rescan until clean. Optional: set **`SNYK_CODE_TEST_ON_EXPORT=1`** so the export hook runs `snyk code test` when the CLI is on `PATH`.

4. **Back to the notebook** — Continue editing the **`.ipynb`**; do not leave behavior only in the exported `.py`.

## Dependencies

- **Before** adding a package (`%pip`, `pip install`, or edits to dependency manifests), run **Snyk `snyk_package_health_check`** (correct `ecosystem`, e.g. `pypi`) for each new package.

- If this repo includes a **verify script** that exports all notebooks and runs Snyk, run it from the repo root (e.g. `SKIP_SNYK=1` to export only when the CLI is unavailable).

## Optional CLI / scripts aligned with a notebook

If the repo has a **script or CLI** that duplicates or wraps notebook training, **keep it aligned** with the notebook: **notebook changes first**, export, scan, then update the script’s flags and defaults only as needed. Document the exact filenames in **this repo’s README** or a short “Project layout” section—not in shared rules copied to other teams.
